> **面向对象**：负责提供摄像头标定参数、RTSP 地址及报警推送地址的上游团队。
> **本文目的**：说明我方系统需要哪些数据项（不论接口格式），帮助上游团队设计数据接口时覆盖所有必要字段。我方可适配上游定义的最终接口形态。

## 接口设计原则

**接口由上游团队主导定义。** 我方系统作为数据消费方，会按照上游最终确定的接口格式进行适配。本文档仅列出我方运行所必需的数据项及其含义——这些数据以什么 JSON 结构、什么字段名、什么传输方式（HTTP API / 配置文件 / 消息队列）来传递，**完全由上游决定**。

---

## ⚠️ 重要前提：地平面假设

本系统采用**平面单应性（Plane Homography）** 模型进行像素 ↔ 世界坐标转换。该模型的**数学前提是：监控场景的地面必须是近似平坦的平面**。

| 场景类型 | 是否可用 | 说明 |
|----------|----------|------|
| 硬化路面、室内地板、广场 | ✅ 适用 | 精度 ~15 cm（0-15m 范围） |
| 轻微起伏（±10 cm 以内） | ⚠️ 勉强可用 | 局部精度下降，远端误差放大 |
| **坡道、斜坡、台阶、坑洼** | ❌ **不可用** | 单应性模型无法描述非平面几何，定位结果完全不可靠 |
| 多层结构（如脚手架、天桥） | ❌ **不可用** | 不同高度的地面不在同一平面上 |

> **为什么？** 单应性矩阵 H 是一个 3×3 的线性变换，它将一个平面上的点映射到另一个平面。如果实际地面不是平面，那么不存在一个统一的 H 能够正确描述画面中所有位置的映射关系——强行拟合会导致远端（或坡面区域）出现几米甚至十几米的定位偏差。

**如果场景确实有坡度怎么办？** 如果必须覆盖有坡度的场景，需要替换为完整的相机模型（PnP），此时需额外提供相机内参矩阵 K（3×3）和畸变系数 D（至少 4 个）。这会改变整个坐标转换管线——请在标定前与我方确认。

### 2.3 H 矩阵约定

**H** 是一个 3×3 单应性矩阵，定义 **OpenCV 像素坐标（左上角原点）→ 中心化 UTM 世界坐标（米）** 的映射：

```
给定像素坐标 (u, v) 和世界均值 (mx, my)：

    [x']   [ h11  h12  h13 ] [ u ]
    [y'] = [ h21  h22  h23 ] [ v ]
    [w']   [ h31  h32  h33 ] [ 1 ]

    world_x = x'/w' + mx
    world_y = y'/w' + my
```

| 约定项 | 说明 |
|--------|------|
| 像素原点 | 画面**左上角**（OpenCV 标准），X→右，Y→下 |
| 世界坐标系 | UTM 米制，经 world_mean 中心化 |
| H 矩阵尺度 | 通常 h33 = 1.0 |
| ⚠️ 前提 | 地面必须近似平坦（见文档顶部警告） |

> 系统会从 H 矩阵**自动推导**相机朝向、相机坐标系基向量等派生量，无需额外提供相机安装角度或俯仰角。

## 4. 标定参数生成方法

### 4.1 原理

采用**平面单应性（Plane Homography）** 模型。**核心前提见文档顶部 ⚠️ 警告**。

当前标定精度（工地硬化路面，23 控制点）：

| 距离范围 | 定位精度 | 适用场景 |
|----------|----------|----------|
| 0 ~ 15 m | ~15 cm ✓ | 人员/车辆精确定位、距离报警 |
| 15 ~ 20 m | ~60 cm ⚠️ | 区域粗略判断、存在性检测 |
| > 20 m | 不建议依赖 | 超出控制点覆盖范围 |

### 4.2 生成步骤

1. **采集控制点**：在画面中选取 ≥ 20 个清晰地面点（均匀分布，远端至少 5 个），记录像素坐标（左上角原点）和 UTM 世界坐标（RTK 采集）。
2. **RANSAC 清洗**：`cv2.findHomography(pixel_pts, world_c, cv2.RANSAC, threshold=1.5)`，自动剔除标注误差大的点。
3. **全量拟合**：用清洗后的内点 + 最小二乘法计算最终 H。
4. **计算 world_mean**：所有有效控制点世界坐标的算术均值。
5. **校核**：检查重投影 RMSE，重点关注远端控制点（误差会被透视放大 3~5 倍）。
6. **地形确认**：确认场景满足地平面假设。

### 4.3 对上游团队的建议

- **标定脚本可参考我方已有实现**：`calibration/compute_calib.py` + `calibration/compute_and_grid_vis.py`。见 [§6](#6-参考实现)。
- **上游只需交付最终结果**（H 矩阵 + world_mean），无需提供原始控制点数据。
- **首次标定前**，请和我方确认每个摄像头场景是否符合地平面假设。

## 5. 坐标系与数学约定

### 5.1 坐标转换链条

```
像素坐标 (u, v)                     ← OpenCV，左上角原点，Y↓
    │  H @ [u, v, 1]ᵀ
    ▼
中心化局部米 (lx, ly)               ← 相对 world_mean
    │  + world_mean
    ▼
UTM 世界米 (world_x, world_y)       ← 绝对地理坐标
    │  local_to_camera() 旋转
    ▼
相机对齐坐标 (cam_x, cam_y)         ← cam_y = 前方, cam_x = 右方
    │
    ▼
网格行列 (row, col)                 ← 占用网格索引
```

### 5.2 自动推导量（无需上游提供）

| 量 | 含义 | 推导方式 |
|----|------|----------|
| `H_inv` | H 的逆矩阵（世界→像素） | `np.linalg.inv(H)` |
| `CAM_FORWARD` | 相机前方单位向量 | 图像中心上方 100px → 世界坐标方向 |
| `CAM_RIGHT` | 相机右方单位向量 | 图像中心右侧 100px → 世界方向（正交化） |
| `CAM_ORIGIN_WORLD` | 相机地面投影位置 | 图像底部中心 → 世界坐标 |

## 6. 参考实现

### 6.1 标定脚本（我方已有）

| 文件 | 功能 | 输入 | 输出 |
|------|------|------|------|
| `calibration/compute_calib.py` | 加载控制点 → RANSAC 剔除离群点 → 视频流叠加验证 | 控制点 txt 文件 + RTSP 流 | 清洗后的控制点 |
| `calibration/compute_and_grid_vis.py` | 从清洗文件计算 H → 画透视网格 → 保存 .npz | 清洗后的控制点 | `camera_calib_371.npz` |

### 6.2 控制点交换格式

标定过程中可参考此格式交换原始控制点：

```
画面左下角0,0
x,y
<像素X>,<像素Y>   <UTM_X>,<UTM_Y>
```

- 像素坐标原点：画面**左下角**（脚本内部自动翻转为 OpenCV 左上角原点）
- UTM 坐标单位：米
- 示例行：`1038,642   812825.46,817636.60`

### 6.3 系统配置加载入口

配置加载集中在 `src/config.py`，当前实现：

```
Config.__init__(camera_id=None)
  ├── 1. 设置硬编码默认值（含 .npz 路径）
  ├── 2. _try_load_json(camera_id)
  │     ├── 读取 config/cameras.mock.json
  │     ├── 匹配 camera_id（默认取 cameras[0]）
  │     └── 覆盖 H, WORLD_MEAN, RTSP_URL, grid, risk 等字段
  │
  └── 3. initialize_calib()
        ├── H 已有值 → 跳过 .npz 加载
        ├── H 为空   → np.load(CALIB_PATH) 回退
        └── 推导 H_inv, CAM_FORWARD, CAM_RIGHT, CAM_ORIGIN_WORLD,
             GRID_COLS/ROWS, BEV_W/H 等（此段逻辑与数据来源无关）
```

> 上游接口就绪后，只需修改步骤 2 的数据来源，其余逻辑不变。

## 附录 A：常见问题

**Q: 最终接口格式什么时候确定？**
A: 由上游团队主导。我方只需要知道数据从哪个渠道（HTTP endpoint / 本地文件 / 消息队列）、以什么格式（JSON / YAML / ...）获取，即可在 `config.py` 中完成适配。**建议上游先给一版 mock 数据（内容不限结构），我方对照调整适配逻辑。**

**Q: 如果摄像头位置或角度变动了怎么办？**
A: 需要重新标定该路摄像头（H 矩阵 + world_mean 更新）。上游接口中更新该路数据即可，我方重启后生效。grid、risk 参数通常无需调整。

**Q: 不使用 UTM 坐标可以吗？**
A: 可以。只要世界坐标系是水平笛卡尔坐标（单位米），模型就能正确工作。world_mean 的值应确保 (world_coord - world_mean) 不会因数值过大导致浮点精度丢失。

**Q: 地面有坡度怎么办？**
A: 平面单应性模型数学上要求地平面平坦。有明显坡面的场景必须换用 PnP 模型，届时需额外提供相机内参矩阵 K 和畸变系数 D。请在标定前与我方确认场景地形。

**Q: 新增一路摄像头需要改代码吗？**
A: 不需要。在 `config/cameras.mock.json` 数组中追加该路配置，重启即可。启动时可通过 `Config(camera_id="372")` 选择目标摄像头。上游接口就绪后同理——上游接口中追加数据即可。
